## Seeded app catalog

Borg reconciles these built-ins:

- `borg:app:codemode-runtime`
- `borg:app:shellmode-runtime`
- `borg:app:memory-system`
- `borg:app:taskgraph-system`
- `borg:app:clockwork-system`
- `borg:app:github`

## Reconciliation lifecycle

Built-ins are reconciled on both:

- `borg init`
- `borg start`

This means missing seeded records are recreated automatically.

## What built-ins represent

Built-ins provide canonical app/capability metadata aligned with core toolchains:

- CodeMode and ShellMode execution capabilities
- Memory and TaskGraph capability sets
- Clockwork scheduling capability set
- GitHub OAuth-based integration capability

## Management constraints

Built-in apps are marked managed/built-in and cannot be deleted via normal app delete endpoint (`403`).

You can still inspect and reason about them through app listing/detail APIs.

## GitHub built-in specifics

GitHub built-in is configured with OAuth2 metadata and expected secrets like:

- `GITHUB_ACCESS_TOKEN`
- `GITHUB_REFRESH_TOKEN`
- `GITHUB_SCOPE`
- `GITHUB_EXPIRES_AT`

These names matter for env projection in execution context.

## Operational takeaway

Treat built-ins as baseline contracts. Extend with custom apps rather than mutating/deleting built-in identity records.
