## Runtime-supported provider kinds

Borg runtime currently resolves these `provider_kind` values:

- `openai`
- `openrouter`
- `lmstudio`
- `ollama`

Any other stored kind is ignored by the active LLM resolver path.

## Kind-specific requirements

Validation rules at provider upsert/model-discovery layer:

- `openai` and `openrouter` require non-empty `api_key`.
- `lmstudio` and `ollama` require non-empty `base_url`.

For local OpenAI-compatible providers (`lmstudio`, `ollama`), runtime may synthesize placeholder API key values when needed.

## OpenAI-compatible behavior

`lmstudio` and `ollama` are resolved through the OpenAI-compatible provider adapter using `{base_url}/v1/*` semantics.

Practical implication: your local endpoint must expose OpenAI-style chat and models endpoints.

## Provider record tips

Keep these fields accurate per provider:

- `provider` (id)
- `provider_kind`
- `enabled`
- `default_text_model`
- `default_audio_model`

Misaligned `provider_kind` values are a common source of silent non-selection.

## Operational checks

After configuring a provider:

1. `GET /api/providers/:provider`
2. `GET /api/providers/:provider/models`
3. Verify provider appears in runtime selection order (see precedence docs).

## CLI note

Older CLI help text may still mention kinds not handled by current runtime resolver. Treat resolver-supported kinds above as the execution source of truth.
