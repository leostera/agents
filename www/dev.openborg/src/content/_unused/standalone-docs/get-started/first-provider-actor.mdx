---
title: Connect Provider, Behavior, and Actor
description: Configure a runnable actor path with explicit provider and behavior records.
---

## 1) Upsert provider

Example (`openai`):

```bash
curl -sS -X PUT http://127.0.0.1:8080/api/providers/openai \
  -H 'content-type: application/json' \
  -d '{
    "provider_kind":"openai",
    "api_key":"YOUR_OPENAI_KEY",
    "enabled":true,
    "default_text_model":"openai/gpt-4.1-mini"
  }'
```

For `lmstudio` or `ollama`, provide `base_url` instead of cloud API key-only shape.

## 2) Create behavior

```bash
curl -sS -X PUT http://127.0.0.1:8080/api/behaviors/borg:behavior:default \
  -H 'content-type: application/json' \
  -d '{
    "name":"default",
    "system_prompt":"You are a pragmatic assistant.",
    "preferred_provider_id":"openai"
  }'
```

## 3) Create actor

```bash
curl -sS -X PUT http://127.0.0.1:8080/api/actors/borg:actor:assistant \
  -H 'content-type: application/json' \
  -d '{
    "name":"assistant",
    "system_prompt":"You are a pragmatic assistant.",
    "default_behavior_id":"borg:behavior:default"
  }'
```

## 4) Validate provider models

```bash
curl -sS http://127.0.0.1:8080/api/providers/openai/models
```

Check that expected model IDs are present and defaults are sensible.

## 5) (Recommended) set runtime default provider

```bash
cargo run -p borg-cli -- providers set default openai
```

This writes runtime preferred provider keys used by resolver ordering.

## Common setup failures

- `400` on provider upsert: missing required `api_key` or `base_url` by kind.
- `400` on actor/behavior upsert: invalid URI fields.
- `404` on model listing: provider disabled or not configured.
