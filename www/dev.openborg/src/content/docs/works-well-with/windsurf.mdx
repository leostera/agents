title: openborg works well with Windsurf
description: Keep Windsurf for rapid implementation and use openborg for durable orchestration boundaries.
---

## Core value split

- **Windsurf**: coding acceleration and rapid implementation loops.
- **openborg**: orchestration reliability for lifecycle-critical execution.

## Best-fit signal

This pairing is strong when your team says:

- "Feature delivery is fast, but handoffs to operational flows are brittle."
- "We need explicit execution boundaries beyond coding context."

## Handoff protocol

1. Implement quickly in Windsurf.
2. Define execution boundaries in openborg (ingress, tasks, review, done).
3. Execute recurring workflows via openborg.
4. Return insights to implementation loop.

## First workflow to implement

Pick one recurring post-implementation operation:

1. define task states and ownership
2. define acceptance criteria per stage
3. capture outputs in task events
4. track cycle-time delta over a month

## Success metrics

- fewer post-implementation execution stalls
- reduced manual coordination between coding and operations
- improved predictability of repeat workflows

## Anti-patterns

- Treating openborg as another coding IDE.
- Running lifecycle-critical workflows without review states.
- Expanding integration scope before one flow is stable.

## When not to pair

If the team is focused only on early coding velocity and has no recurring operations yet.
