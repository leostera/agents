## Short answer

- Choose **Tess AI** first if your top priority is model marketplace breadth and routing optionality.
- Choose **openborg** first if your top priority is explicit execution lifecycle and operational repeatability.

## Core difference

Tess AI optimizes *which model/provider to use*.
openborg optimizes *how workflows execute over time*.

## Best-fit by decision axis

| Decision axis | Better primary fit | Why |
| --- | --- | --- |
| Broad provider/model experimentation | Tess AI | Model-routing breadth is central. |
| Durable multi-stage workflow control | openborg | Task/session semantics are central. |
| Fast model switching by cost/latency | Tess AI | Product optimized for this choice set. |
| Explicit reviewable execution lifecycle | openborg | Runtime state model is explicit. |

## Honest tradeoffs

Where Tess AI is stronger:

- provider optionality and model comparison workflows
- lower friction for model-market experimentation

Where openborg is stronger:

- durable workflow state and governance
- better fit when failure/recovery behavior matters as much as model choice

## Migration path (Tess AI -> openborg)

1. Keep Tess AI where model routing is still the main differentiator.
2. Move one high-value repeated workflow into openborg.
3. Add explicit task states and review gate.
4. Keep model selection concerns separate from execution-governance concerns.

## Coexistence pattern

- Tess AI: model-selection layer.
- openborg: workflow-execution layer.

This split works when model strategy and runtime strategy are both important.

## 14-day evaluation plan

1. Pick one model-sensitive experiment and one lifecycle-sensitive operation.
2. Run model-sensitive path with Tess AI baseline.
3. Run lifecycle path with openborg baseline.
4. Compare rework rate, operator overhead, and state visibility.
