## Purpose

This map keeps content consistent across the three primary surfaces:

1. marketing (`/`)
2. standalone user docs (`/docs/`)
3. platform dev docs (`/platform/`)

## Primary narrative arc

1. **Problem**: prompt-only workflows break under operational load.
2. **Mechanism**: session-first ingress + explicit task lifecycle + port contracts.
3. **Proof**: docs depth, changelog cadence, clear caveat disclosures.
4. **Decision**: comparison and coexistence guidance.
5. **Activation**: fast get-started path plus first real workflow setup.

## Information architecture by intent

### Learn intent (user docs)

User asks: "How does this system work?"

Pages to route:

- `Standalone Borg Docs`
- `Learn`
- `Providers`
- `Models`

### Build intent (user docs)

User asks: "How do I run this now?"

Pages to route:

- `Get Started`
- `Integrations`
- `Apps`

### Build intent (platform docs)

User asks: "How do I build on top of borg?"

Pages to route:

- `Platform Dev Docs`
- `APIs`
- `Extending Borg`
- `Knowledge Base`

### Buy/adopt intent

User asks: "Why this vs alternatives?"

Pages to route:

- `Comparisons`
- `Works Well With`
- `Changelog`

## Tone rules by surface

- **Landing hero**: concrete outcome language, no architecture jargon.
- **Docs reference**: precise runtime semantics and caveats.
- **Comparisons**: decision-grade, honest about where others win.
- **Works Well With**: role split and handoff protocol, not competition framing.

## Required conversion assets

Each major section should expose:

1. one practical first step
2. one realistic limitation/caveat
3. one path to deeper technical detail

This keeps trust high while still driving action.
