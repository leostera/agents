## API domains

Borg REST surface groups into these domains:

- system + ingress
- providers/models
- apps/auth/connections/secrets
- actors/behaviors/sessions/users
- ports + bindings + context
- taskgraph
- clockwork
- memory
- observability/devmode/fs

## High-traffic routes

### System and ingress

- `GET /health`
- `POST /ports/http`
- `POST /ports/http/audio`

### Providers and models

- `GET /api/providers`
- `GET|PUT|DELETE /api/providers/:provider`
- `GET /api/providers/:provider/models`
- `POST /api/providers/openai/device-code/start`

### Apps

- `GET /api/apps`
- `GET|PUT|DELETE /api/apps/:app_id`
- `GET|PUT|DELETE /api/apps/:app_id/capabilities/:capability_id`
- `GET|PUT|DELETE /api/apps/:app_id/connections/:connection_id`
- `GET|PUT|DELETE /api/apps/:app_id/secrets/:secret_id`
- `POST /api/apps/:app_id/oauth/start`
- `GET /oauth/:provider/callback`

### Actors/behaviors/sessions/users

## Error shape and usage

Most controllers return JSON error payloads with status-appropriate codes (`400`, `404`, `500`, `502`).

For client SDKs, treat URI parsing and required-field checks as first-class pre-validation to reduce noisy retries.

## Practical integration order

1. Use `health` and HTTP port routes first.
2. Configure provider + actor/behavior.
3. Add app connections/secrets.
4. Introduce TaskGraph/Clockwork once core turns are stable.
5. Layer connector ports last.

## Compatibility note

REST is the currently active production integration surface. GraphQL crate exists but is not mounted in active API router yet.
