## Current status

Borg includes a substantial standalone GraphQL crate at `crates/borg-gql`, but active `borg-api` routing does not currently expose `/graphql` or `/graphiql`.

Use REST as the production integration surface today.

## What exists in `borg-gql`

- Typed schema and resolver context over Borg DB + memory.
- Broad entity coverage (actors, behaviors, sessions/messages, providers, apps, taskgraph, clockwork, memory, policies, users).
- Generated schema artifact (`schema.graphql`) in crate.

This means most schema design work exists, but transport mounting is pending in runtime server.

## Known limitations in standalone mode

Certain runtime-wrapper mutations currently return explicit placeholder errors in standalone crate context, including chat/port wrapper paths (`runActorChat`, `runPortHttp`).

Do not treat standalone crate as drop-in runtime control plane yet.

## Why docs still include GraphQL section

GraphQL is clearly on the architecture path (see `docs/rfds/RFD0023-borg-gql.md`) and clients/packages already reference it. Keeping this section explicit prevents false assumptions while preserving migration context.

## Safe usage guidance

- For runtime integration: use REST routes.
- For schema exploration/planning: inspect `crates/borg-gql/schema.graphql`.
- For future migration: track `borg-api` router integration work before exposing GraphQL to users.
