---
title: openborg works well with v0
description: Keep v0 for rapid interface generation and use openborg for repeatable backend/process orchestration.
---

## Core value split

- **v0**: rapid UI and app-surface generation.
- **openborg**: lifecycle-governed workflow execution across systems.

## Best-fit signal

This pairing is strong when your team says:

- "Frontend generation is fast, but operational flows behind it are brittle."
- "We need durable process control, not just fast scaffolding."

## Handoff protocol

1. Iterate interface and product surface in v0.
2. Move recurring backend/process flows into openborg.
3. Expose status/results back to product surface as needed.
4. Keep v0 for high-speed UI iteration.

## First workflow to implement

Pick one user-triggered flow with repeated operational cost.

1. define request/response contract
2. map execution stages in TaskGraph
3. add review checkpoint for risky actions
4. track completion reliability

## Success metrics

- fewer user-visible failures in repeated flows
- improved observability of in-flight execution state
- faster debugging of cross-system failures

## Anti-patterns

- Treating v0 as orchestration runtime.
- Treating openborg as UI generation engine.
- Shipping without explicit state transitions.

## When not to pair

If your workload is still pure UI exploration and no repeated operational path exists.
