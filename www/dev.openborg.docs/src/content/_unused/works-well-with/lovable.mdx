title: openborg works well with Lovable
description: Use Lovable for concept-to-product speed and openborg for execution reliability once workflows repeat.
---

## Core value split

- **Lovable**: fast product concept and UX iteration.
- **openborg**: explicit runtime lifecycle for repeatable execution.

## Best-fit signal

This pairing is strong when your team says:

- "Discovery speed is great, but delivery operations are not predictable."
- "We need a controlled execution layer behind product iteration."

## Handoff protocol

1. Run discovery and product iteration in Lovable.
2. Identify recurring workflows that affect reliability.
3. Move those workflows into openborg with task/review structure.
4. Keep Lovable focused on user-facing iteration.

## First workflow to implement

Select one repeat operational sequence tied to product growth.

1. define workflow trigger and output contract
2. model stages in openborg
3. require review before terminal state
4. track rework reduction

## Success metrics

- fewer repeated execution regressions
- better handoff clarity from product to operations
- faster root-cause isolation when runs fail

## Anti-patterns

- Forcing discovery loops into heavy orchestration too early.
- Blurring ownership between product iteration and runtime execution.
- Expanding scope before first workflow has stable metrics.

## When not to pair

If you are still validating problem/solution fit and no operational workflow repeats yet.
