## Resolution algorithm (runtime truth)

For each inbound message, Borg runs this sequence:

1. Compute `conversation_key` for the port.
2. Resolve session via `resolve_port_session(port, conversation_key, requested_session)`.
3. Resolve actor via `resolve_port_actor(port, conversation_key, requested_actor, default_actor)`.
4. Ensure canonical session row exists (`upsert_session`).
5. Persist port-session context snapshot when present.
6. Execute supervisor call with resolved `actor_id` and `session_id`.

This flow is shared by HTTP ingress and managed connectors through the bridge loop.

## Conversation key source by port

- **HTTP**: `session_id` if provided; otherwise `user_key`.
- **Telegram**: `telegram:conversation:<chat_id>`.
- **Discord**: `discord:conversation:<channel_id>`.

The key is what gives you stable session continuity per channel conversation.

## Actor binding precedence

`resolve_port_actor` follows strict precedence:

1. `requested_actor_id` (if supplied) and persist binding.
2. Existing binding for `port + conversation_key`.
3. Port default actor (`assigned_actor_id`) and persist binding.
4. Error if none is available.

For HTTP, if resolution fails, request fails. For managed connectors, bridge loop warns and falls back to `session_id` as actor id.

## Slash command behavior

Ingress supports command routing before normal chat:

- `/help`
- `/start`
- `/model [name]`
- `/participants`
- `/context`
- `/reset`
- `/compact`

Some commands are local replies (`/help`, `/start`), others forward to runtime control commands.

## Session context snapshots

Per-port session context is stored in `port_session_ctx` and exposed via:

- `GET /api/ports/:port_uri/sessions/:session_id/context`
- `PUT /api/ports/:port_uri/sessions/:session_id/context`
- `GET /api/sessions/:session_id/context` (any-port lookup)

This is how commands like `/participants` and `/context` can reference channel metadata.

## Port control surface

Control-plane endpoints:

- `/api/ports`
- `/api/ports/:port_uri`
- `/api/ports/:port_uri/settings/*`
- `/api/ports/:port_uri/bindings/*`
- `/api/ports/:port_uri/actor-bindings/*`
- `/api/ports/:port_uri/sessions/:session_id/context`

`port_uri` must follow `borg:port:<name>`.

## Common failure modes

- Invalid URI fields (`user_key`, `session_id`, `agent_id`) => `400`.
- Empty text payload => `400`.
- Unknown command => `400` on HTTP command parsing path.
- Missing actor binding with no default => runtime binding error.

## Practical debugging checklist

1. Confirm `conversation_key` is stable across turns.
2. Check binding rows under `/api/ports/:port_uri/bindings`.
3. Check actor bindings under `/api/ports/:port_uri/actor-bindings`.
4. Confirm the target actor exists and is valid.
5. Verify session context exists if command output depends on it.
