## Auth strategies

Current app auth strategies in active flow:

- `none`
- `oauth2`

`oauth2` strategy uses app auth config metadata (`authorize_url`, `token_url`, scopes, client id/secret, userinfo URL).

## OAuth lifecycle

OAuth start endpoint:

- `POST /api/apps/:app_id/oauth/start`

Callback endpoint:

- `GET /oauth/:provider/callback`

During OAuth start, Borg stores connection state with:

- CSRF state
- PKCE verifier
- pending connection status (`pending_oauth`)

On successful callback + token exchange, connection is updated to `connected` and tokens/secrets are persisted.

## Connection model

Connections attach identity and auth state to an app, optionally scoped to owner user.

Runtime secret projection prefers:

1. connected connection owned by current user
2. otherwise connected shared connection (no owner)

## Secrets model

Secrets are stored as app secret records and may be projected into CodeMode env only when their keys match app `available_secrets` names (with practical alias matching rules).

Example pattern:

- stored key: `access_token`
- projected env key: `GITHUB_ACCESS_TOKEN`

## Security posture

Treat secrets as production credentials:

- restrict who can write/read app secret endpoints
- avoid logging secret payloads
- keep `available_secrets` minimal and explicit
- rotate OAuth credentials as needed

## Operational checks

1. Verify app auth strategy/config.
2. Start OAuth and inspect pending connection.
3. Complete callback and confirm connection status `connected`.
4. Confirm expected secrets exist for that connection.
5. Verify projected env keys in controlled execution path.
