---
title: HTTP Port
description: Programmatic ingress contract for text and audio turns.
---

## Endpoints

- `POST /ports/http`
- `POST /ports/http/audio`

Both return `session_id`, `reply`, and `tool_calls` on success, and set `x-borg-session-id` response header.

## Text request contract

Required fields:

- `user_key` (URI)
- `text`

Optional fields:

- `session_id` (URI)
- `agent_id` (URI, actor-like value expected)
- `metadata`

Validation failures return `400` (invalid URI or empty text).

## Audio request contract

Required fields:

- `user_key` (URI)
- `audio_base64`

Optional fields:

- `mime_type`
- `duration_ms`
- `language_hint`
- `session_id`
- `agent_id`
- `metadata`

Invalid/empty base64 returns `400`.

## Session + actor resolution behavior

For HTTP ingress:

- conversation key = `session_id` if present, else `user_key`
- actor binding precedence = requested -> existing -> port default

If actor binding cannot be resolved, request fails.

## Slash command support

HTTP port supports command parsing for:

- `/help`
- `/start`
- `/model`
- `/participants`
- `/context`
- `/reset`
- `/compact`

Unknown commands are rejected.

## Response usage pattern

For robust clients:

1. Capture `x-borg-session-id` on first successful response.
2. Send that value as `session_id` on follow-up turns.
3. Persist `session_id` per conversation thread in your client app.

## Caveat: metadata payload

`metadata` is accepted in HTTP payload shape but is currently not a primary driver of HTTP execution routing. Do not depend on it for critical behavior without validating your runtime version.
