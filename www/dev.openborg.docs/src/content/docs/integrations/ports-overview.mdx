## What a port is

A port is an ingress adapter that turns external input into Borg runtime messages.

Every port ultimately feeds the same session-turn pipeline:

- normalize conversation identity
- resolve session/actor bindings
- execute turn through supervisor

## Port families

Current runtime ingress paths:

- HTTP text: `POST /ports/http`
- HTTP audio: `POST /ports/http/audio`
- Managed connectors: Telegram and Discord

Unknown provider types can remain configured in DB but are not actively started by supervisor.

## Shared invariants across ports

- Session resolution is keyed by `port + conversation_key`.
- Actor resolution uses requested/existing/default precedence.
- Session row is upserted before execution.
- Port session context snapshots may be persisted for diagnostics/commands.

## Control-plane routes

Use `/api/ports*` routes to manage:

- ports and settings
- conversation bindings
- actor bindings
- per-session context snapshots

`port_uri` must be `borg:port:<name>`.

## Privacy model

Ports can be configured as guest-allowing or restricted.

For private Telegram/Discord ports, allow-list configuration controls who is accepted from external channels.

## Operational recommendation

Start with HTTP ingress for initial validation, then add managed connectors once session/actor behavior is stable.
